General troubleshooting guide for Sahara's feedback pipeline, AI health monitoring, and scheduled jobs.

## Feedback Pipeline

### Pipeline Flow

```
User Action (thumbs up/down, chat, event survey)
        |
        v
API Route (/api/feedback/signal, /api/feedback/event, /api/fred/chat)
        |
        +---> Idempotency Check (AI-4120)
        |         - signal: user_id + message_id + signal_type
        |         - event: user_id + event_name (1hr window)
        |         - insight: title + type (4hr window)
        |
        +---> Consent + Throttle Check
        |
        +---> Insert feedback_signals
        |
        v
Clustering Pipeline (processes signals with status='new')
        |
        +---> getUnprocessedSignals() — batch of 500
        +---> AI clustering + pattern detection
        +---> upsertInsightWithSignals() — hash-based dedup
        +---> markSignalsProcessed()
        |
        v
feedback_insights (patterns, trends, anomalies)
        |
        +---> Admin dashboard
        +---> Linear issue creation (if severity >= high)
        +---> RLHF prompt patches (Phase 76)
```

### Common Pipeline Issues

#### Signals stuck in "new" status
```sql
-- Check backlog
SELECT COUNT(*) FROM feedback_signals WHERE processing_status = 'new';

-- Check oldest unprocessed
SELECT id, created_at FROM feedback_signals
WHERE processing_status = 'new'
ORDER BY created_at ASC LIMIT 5;
```

**Fix:** The clustering pipeline may have failed. Check the cron/trigger that runs it. Manually re-trigger if needed.

#### Duplicate signals
Handled by AI-4120 idempotency guards:
- Unique partial index: `(user_id, message_id, signal_type) WHERE message_id IS NOT NULL`
- Application-level check-then-insert in `insertFeedbackSignal`
- Race condition handling via PostgreSQL error code 23505

#### Duplicate insights
- `upsertInsightWithSignals` uses `cluster_embedding_hash` for dedup within 4hr window
- `insertFeedbackInsight` deduplicates by `title + insight_type` within 4hr window

## AI Provider Health

### Circuit Breaker States

| State | Meaning | Action |
|-------|---------|--------|
| `closed` | Healthy, requests flowing | None |
| `half-open` | Testing recovery | Monitor - will auto-resolve or re-open |
| `open` | Provider down, traffic redirected | Check provider status page |

### Provider Status Pages
- **Anthropic:** https://status.anthropic.com
- **OpenAI:** https://status.openai.com
- **Google AI:** https://status.cloud.google.com

### Provider Failover
The system uses a fallback chain: Primary -> Fallback1 -> Fallback2. If the primary provider's circuit breaker opens, requests automatically route to the next healthy provider.

## Alert Thresholds

| Metric | Warning | Critical |
|--------|---------|----------|
| Error Rate | 5% | 10% |
| P95 Latency | 2000ms | 5000ms |
| Traffic Distribution | < 10% expected | N/A |

Alerts are generated by `lib/monitoring/alerts.ts` and delivered via `lib/monitoring/alert-notifier.ts` (Slack, PagerDuty, email).

Alert scheduler runs every 15 minutes by default (`lib/monitoring/alert-scheduler.ts`).

## GDPR / Data Retention

Feedback signals have an `expires_at` field. Expired data is cleaned up by `deleteExpiredFeedback()`.

To manually purge:
```sql
-- Check what would be deleted
SELECT COUNT(*) FROM feedback_signals
WHERE expires_at < now() AND expires_at IS NOT NULL;

-- Delete (handled by cron normally)
DELETE FROM feedback_signals
WHERE expires_at < now() AND expires_at IS NOT NULL;
```

To delete all feedback for a specific user (GDPR right to erasure):
```typescript
import { deleteFeedbackForUser } from "@/lib/db/feedback";
await deleteFeedbackForUser(userId);
```

## Escalation

1. **Auto-retry:** Most pipeline jobs retry automatically (Trigger.dev: 2 attempts, Vercel cron: built-in)
2. **Alert notification:** Critical alerts go to configured channels (Slack/PagerDuty/email)
3. **Manual check:** Review Trigger.dev dashboard, Vercel logs, Supabase query console
4. **Code fix:** File Linear issue with `pipeline` label in Sahara project
