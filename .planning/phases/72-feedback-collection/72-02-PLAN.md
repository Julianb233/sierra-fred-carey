2. Import `MAX_DETAILED_FEEDBACK_PER_WEEK` from `@/lib/feedback/constants` (value: 1).
3. Query `feedback_signals` table for signals from this user where:
   - `created_at` >= 7 days ago (use `new Date(Date.now() - 7 * 24 * 60 * 60 * 1000).toISOString()`)
   - `category IS NOT NULL` OR `comment IS NOT NULL` (these are "detailed" signals — basic thumbs have null for both)

Use `.select('created_at', { count: 'exact' })` with `.order('created_at', { ascending: true }).limit(1)` to efficiently get both count and earliest timestamp in one query.

"Detailed feedback" means a signal that has either a category or a comment. Basic thumbs-only signals (rating set, category=null, comment=null) bypass throttling entirely.

  <action>
Create `app/api/feedback/signal/route.ts` following the same pattern as `app/api/feedback/consent/route.ts`.

**Request body shape:**
```typescript
{
  message_id: string       // required — the FRED message being rated
  signal_type: 'thumbs_up' | 'thumbs_down'  // required
  rating: 1 | -1           // required — 1 for up, -1 for down
  category?: string | null  // optional — from FEEDBACK_CATEGORIES
  comment?: string | null   // optional — free-text (max 500 chars)
  session_id?: string | null // optional — chat session ID if available
}
```

**Route implementation (POST only):**

1. **Auth:** Use `createClient()` (await) to get user via `supabase.auth.getUser()`. Return 401 if no user.

2. **Validate body:** Parse JSON. Validate:
   - `message_id` is a non-empty string
   - `signal_type` is one of `SIGNAL_TYPES` ('thumbs_up' | 'thumbs_down')
   - `rating` is 1 or -1
   - If `category` provided, it must be in `FEEDBACK_CATEGORIES`
   - If `comment` provided, truncate to 500 chars
   - Return 400 with specific error message on validation failure

3. **Consent check:** Use `createServiceClient()` and call `getUserConsentStatus(serviceClient, user.id)`. If false, return 403 with `{ error: "Feedback consent required", code: "CONSENT_REQUIRED" }`. The client can use this code to show the consent banner.

4. **Throttle check (conditional):** Only check throttle if this is a "detailed" signal (category or comment is present and non-null). Call `checkDetailedFeedbackThrottle(user.id)`. If not allowed, return 429 with `{ error: "Feedback limit reached", nextAllowedAt, code: "THROTTLED" }`. Basic thumbs-only signals (no category, no comment) SKIP throttling entirely.

5. **Determine tier and weight:** Fetch user tier. Use pattern from codebase: query `subscriptions` table or use `getUserSubscription()` if available. Fallback to 'free' tier. Look up weight from `TIER_WEIGHTS` constant (`{ free: 1.0, pro: 3.0, studio: 5.0 }`).

6. **Build signal insert:** Construct `FeedbackSignalInsert` object:
   ```typescript
   {
     user_id: user.id,
     session_id: body.session_id ?? null,
     message_id: body.message_id,
     channel: 'chat' as const,
     signal_type: body.signal_type,
     rating: body.rating,
     category: body.category ?? null,
     comment: body.comment?.slice(0, 500) ?? null,
     sentiment_score: null,        // populated later by sentiment pipeline
     sentiment_confidence: null,
     user_tier: tier,
     weight: TIER_WEIGHTS[tier],
     consent_given: true,          // already verified above
     expires_at: calculateExpiryDate(),
     metadata: {},
   }
   ```

7. **Apply consent and insert:** Use `applyConsent()` from consent.ts to stamp consent fields (redundant but defensive). Call `insertFeedbackSignal(signal)` from `@/lib/db/feedback`.

8. **Return 201** with `{ success: true, id: data.id }`. Wrap the insert in try/catch — on error, log and return 500.

**Performance consideration:** The route itself is synchronous (awaits the insert). The FIRE-AND-FORGET happens on the CLIENT side (the useFeedback hook from Plan 01 does not await the fetch). The server insert is fast (<10ms for a single row) so the total p95 should be well under 50ms.

**Do NOT add rate limiting via Upstash here.** The throttle utility handles the business-logic throttle. HTTP-level rate limiting is handled by the existing middleware if needed.

For the user tier lookup, use this approach:
```typescript
const { data: sub } = await serviceClient
  .from('subscriptions')
  .select('tier')
  .eq('user_id', user.id)
  .single()
const tier = (sub?.tier as 'free' | 'pro' | 'studio') || 'free'
```
