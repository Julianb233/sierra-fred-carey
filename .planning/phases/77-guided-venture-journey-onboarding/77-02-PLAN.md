  **3. Update `middleware.ts`:**

  Add a post-auth redirect check for the welcome flow. After the existing auth check (around line 41-47), add logic to redirect authenticated users to `/welcome` if they haven't completed the intake.

  IMPORTANT: This redirect MUST NOT apply to:
  - The `/welcome` route itself (infinite redirect loop)
  - API routes (`/api/*`)
  - Auth routes (`/login`, `/signup`, `/get-started`)
  - Static assets and public routes
  - The `/onboarding` route (legacy flow)

  It MUST apply to:
  - `/dashboard` and all sub-routes
  - `/chat` and `/agents`
  - Any protected route where the user would interact with the product

  Implementation approach -- add this block AFTER the existing `isProtectedRoute` check (around line 47):

  ```typescript
  // Welcome flow enforcement: redirect users who haven't completed intake
  // Only check on dashboard/product routes, not on /welcome itself or API routes
  const welcomeExemptPaths = ['/welcome', '/api/', '/login', '/signup', '/get-started', '/onboarding', '/_next/', '/favicon']
  const needsWelcomeCheck = user && !welcomeExemptPaths.some(p => pathname.startsWith(p)) && isProtectedRoute(pathname)

  if (needsWelcomeCheck) {
    // Import createClient at the top of the file if not already imported
    // Use the Supabase server client from the already-created session
    // We need to check the profile without making the middleware too slow
    // Strategy: Check a cookie/header first, fall back to DB query

    // For now, use a lightweight DB check. The query is indexed (idx_profiles_journey_welcomed).
    // This adds ~5-10ms to middleware on protected routes.
    const { createClient: createServerClient } = await import("@/lib/supabase/server")
    const supabase = await createServerClient()
    const { data: profile } = await supabase
      .from("profiles")
      .select("journey_welcomed")
      .eq("id", user.id)
      .single()

    if (profile && profile.journey_welcomed === false) {
      const welcomeUrl = new URL("/welcome", request.url)
      const redirectResponse = NextResponse.redirect(welcomeUrl)
      redirectResponse.headers.set("X-Request-ID", requestId)
      return redirectResponse
    }
  }
  ```

  CRITICAL: The `user` variable comes from the existing `const { response, user } = await updateSession(request)` call on line 30. Do NOT create a new auth check -- reuse the existing one.

  CRITICAL: The `createServerClient` import must be dynamic (inside the if block) since middleware.ts uses edge runtime and this avoids loading the server client unnecessarily on non-protected routes.

  CRITICAL: If the profile query fails or returns null (e.g., profile hasn't been created yet by the trigger), do NOT redirect to /welcome. Let the user through -- the welcome page itself handles the auth check. Only redirect when `journey_welcomed === false` explicitly.

  IMPORTANT: The existing middleware file gets auto-renamed to proxy.ts during build (noted in CLAUDE.md). This is a known Next.js 16 behavior. Do NOT rename the file -- keep it as `middleware.ts`. The build process handles the rename automatically.
