**lib/rlhf/patch-manager.ts** — Patch lifecycle management.

Imports:
- `sql` from `@/lib/db/supabase-sql`
- Types from `@/lib/rlhf/types`
- `logger` from `@/lib/logger`

Functions:

1. **`getPatchById(id: string): Promise<PromptPatch | null>`**
   - SELECT from prompt_patches WHERE id
   - Map DB columns to PromptPatch type (snake_case → camelCase)

2. **`getPatchesByStatus(status: PatchStatus | PatchStatus[]): Promise<PromptPatch[]>`**
   - SELECT from prompt_patches WHERE status IN (...)
   - ORDER BY created_at DESC

3. **`approvePatch(patchId: string, approvedBy: string): Promise<PromptPatch>`**
   - Verify patch exists and status is 'draft'
   - UPDATE status = 'approved', approved_by, approved_at = NOW()
   - Return updated patch
   - Throw if patch not found or wrong status

4. **`rejectPatch(patchId: string, reason?: string): Promise<PromptPatch>`**
   - Verify patch exists and status is 'draft' or 'approved'
   - UPDATE status = 'rejected', metadata.rejectionReason = reason
   - Return updated patch

5. **`activatePatch(patchId: string): Promise<PromptPatch>`**
   - Verify patch exists and status is 'approved'
   - UPDATE status = 'active', activated_at = NOW()
   - Return updated patch
   - This makes the patch available to buildPromptWithSupplements

6. **`deactivatePatch(patchId: string): Promise<PromptPatch>`**
   - Verify patch exists and status is 'active'
   - UPDATE status = 'archived', deactivated_at = NOW()
   - Return updated patch

7. **`launchPatchAsTest(patchId: string, experimentConfig: { name: string, description?: string, trafficPercentage?: number }): Promise<{ patch: PromptPatch, experimentId: string }>`**
   - Verify patch exists and status is 'approved'
   - Create a new AB experiment via sql INSERT into ab_experiments
   - Create two variants: 'control' (no patch, 50%) and 'treatment' (with patch, 50%)
   - UPDATE prompt_patches: status = 'testing', experiment_id = new experiment ID
   - Return the patch and experiment ID

8. **`getActiveDBPatches(topic?: string): Promise<PromptPatch[]>`**
   - SELECT from prompt_patches WHERE status = 'active'
   - Optionally filter by topic
   - Cached for 60 seconds to avoid per-request DB calls
   - Used by buildPromptWithSupplements for runtime patch loading

**lib/rlhf/patch-tracker.ts** — Tracks thumbs-up improvement after patch deployment.

Imports:
- `sql` from `@/lib/db/supabase-sql`
- Types from `@/lib/rlhf/types`
- `logger` from `@/lib/logger`

Constants:
```typescript
export const TRACKING_WINDOW_DAYS = 14 // 2-week tracking window (REQ-R5)
```

Functions:

1. **`startPatchTracking(patchId: string, topic: string): Promise<void>`**
   - Query current thumbs-up ratio for the topic (baseline):
     ```sql
     SELECT 
       COUNT(*) FILTER (WHERE rating = 1) as thumbs_up,
       COUNT(*) FILTER (WHERE rating = -1) as thumbs_down
     FROM feedback_signals
     WHERE created_at > NOW() - INTERVAL '14 days'
       AND metadata->>'topic' = $topic
     ```
   - UPDATE prompt_patches SET performance_metrics = { baselineThumbsRatio, trackingSince: NOW(), trackingUntil: NOW() + 14 days }

2. **`getPatchPerformance(patchId: string): Promise<{ baseline: number, current: number, improvement: number, daysRemaining: number, significant: boolean } | null>`**
   - Get the patch and its performance_metrics
   - Query current thumbs ratio for the same topic since trackingSince
   - Compute improvement = (current - baseline) / baseline * 100
   - Check if tracking window has elapsed
   - Determine significance using chi-squared test from lib/statistics/significance.ts

3. **`checkPatchImprovement(patchId: string): Promise<'improving' | 'neutral' | 'degrading' | 'insufficient_data'>`**
   - Calls getPatchPerformance
   - Returns 'improving' if improvement > 5% and significant
   - Returns 'degrading' if improvement < -5% and significant
   - Returns 'neutral' if not significant
   - Returns 'insufficient_data' if < 100 signals in tracking window

4. **`updateAllPatchMetrics(): Promise<void>`**
   - Get all patches with status = 'active' or 'testing'
   - For each, call getPatchPerformance and update performance_metrics
   - Called by daily Trigger.dev job

Update `buildPromptWithSupplements` to load active patches from the database at runtime.

Changes to lib/ai/prompt-layers.ts:

1. Add import: `import { getActiveDBPatches } from "@/lib/rlhf/patch-manager"`

2. Add a new async function `loadActivePatches`:
   ```typescript
   /**
    * Load active patches from DB and convert to SupplementalPromptPatch format.
    * Cached internally by getActiveDBPatches (60s TTL).
    */
   export async function loadActivePatches(): Promise<SupplementalPromptPatch[]> {
     try {
       const dbPatches = await getActiveDBPatches()
       return dbPatches.map(p => ({
         id: p.id,
         content: p.content,
         source: p.source,
         sourceId: p.sourceId ?? undefined,
         active: true,
         createdAt: p.createdAt,
       }))
     } catch (err) {
       // Fail open — if DB is unavailable, continue without patches
       console.error('[prompt-layers] Failed to load active patches:', err)
       return []
     }
   }
   ```

3. Add a new async variant: `buildPromptWithSupplementsAsync`:
   ```typescript
   /**
    * Async version that loads active patches from DB.
    * Use this in production chat routes.
    * Falls back to in-memory SUPPLEMENTAL_PATCHES if DB load fails.
    */
   export async function buildPromptWithSupplementsAsync(
     founderContext: string
   ): Promise<string> {
     const dbPatches = await loadActivePatches()
     const allPatches = [...SUPPLEMENTAL_PATCHES, ...dbPatches]
     return buildPromptWithSupplements(founderContext, allPatches)
   }
   ```

4. Keep the existing synchronous `buildPromptWithSupplements` unchanged for backward compatibility.

5. Export `loadActivePatches` and `buildPromptWithSupplementsAsync`.

Note: Do NOT modify `FRED_CORE_PROMPT` — it remains immutable. Patches are appended after the core prompt during assembly.
