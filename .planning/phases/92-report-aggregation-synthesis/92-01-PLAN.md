Create `lib/report/step-mapping.ts` as the single source of truth for which journey steps map to which report sections.

Define the 5 report sections and 19 steps from PROJECT.md lines 28-33:

```
Section 1: "Your Core Offer" (4 steps)
  - Step 1: Define your product/service → maps to journey step c_one_sentence (or c_problem_statement as fallback)
  - Step 2: Identify your ideal customer → maps to v_icp_defined
  - Step 3: Describe the problem in their words → maps to c_problem_statement (or c_who_has_problem)
  - Step 4: Set your pricing and explain why → maps to v_willingness_pay (or v_competitor_pricing)

Section 2: "Your Founder Story" (3 steps)
  - Step 5: Your personal why → maps to c_define_why
  - Step 6: Your unfair advantage → maps to c_unfair_advantages
  - Step 7: 30-second elevator pitch → maps to v_positioning_statement (or c_one_sentence)

Section 3: "Unit Economics" (4 steps)
  - Step 8: Customer acquisition cost (CAC) → maps to v_willingness_pay (cost side) — manual step
  - Step 9: Lifetime value (LTV) → manual step
  - Step 10: LTV:CAC ratio → manual step
  - Step 11: Path to profitability → manual step

Section 4: "Scaling Operations" (4 steps)
  - Step 12: #1 process that breaks at scale → manual step
  - Step 13: Bottleneck automation plan → manual step
  - Step 14: Core playbooks → manual step
  - Step 15: Repeatable acquisition channel → maps to v_distribution_channels

Section 5: "Leadership Mindset" (4 steps)
  - Step 16: Delegate something you love → manual step
  - Step 17: The hard conversation you need to have → manual step
  - Step 18: Your support system → maps to c_cofounder_status
  - Step 19: Your leadership style → maps to c_mindset_check (or c_strengths)
```

For each step, define:
- `id`: string identifier (e.g., "core-offer-product")
- `sectionId`: which section it belongs to
- `label`: display label from PROJECT.md
- `description`: what this step captures
- `journeyStepIds`: array of oases_progress step_id values to pull answers from (primary + fallbacks). Some steps may have NO matching journey step (new steps for v9.0 that founders answer during the journey). For those, use an empty array — they will be populated as founders progress.
- `sortOrder`: position within the section (1-based)

Export types:
- `ReportStepDef` — shape of a step definition
- `ReportSectionDef` — shape of a section (id, title, description, steps[])
- `REPORT_SECTIONS` — array of 5 ReportSectionDef objects
- `REPORT_STEPS` — flat array of all 19 ReportStepDef objects
- `getStepsBySection(sectionId)` — helper returning steps for a section

IMPORTANT: The mapping from 19 report steps to 120 journey steps is a best-effort mapping. Some report steps (especially Unit Economics and Scaling Operations) may not have direct journey step matches yet. This is expected — the aggregator handles missing data gracefully.

Use descriptive section IDs: "core-offer", "founder-story", "unit-economics", "scaling-operations", "leadership-mindset".

<task type="auto">
  <name>Task 2: Create aggregator.ts — pull and structure founder answers from the database</name>
  <files>lib/report/aggregator.ts</files>
  <action>
Create `lib/report/aggregator.ts` that:

1. Takes a `userId: string` parameter
2. Queries `oases_progress` for ALL rows where `user_id = userId` (single query, not per-step)
3. Builds a lookup map: `stepId → { answer: string, distilled: string }` from the `metadata` JSONB column
4. For each of the 19 report steps (from step-mapping.ts):
   - Look up the primary journeyStepId in the map
   - If found, use `metadata.distilled` (preferred) or `metadata.answer` as the answer text
   - If primary not found, try fallback journeyStepIds
   - If none found, set answer to `null`
5. Also query the `profiles` table for `company_name` (or `venture_name`), `full_name` (or `name`), and `industry` — needed for report context
6. Return structured data matching this shape:

```typescript
export interface AggregatedStep {
  stepId: string           // report step ID (e.g., "core-offer-product")
  label: string            // display label
  answer: string | null    // founder's answer text (distilled preferred)
  sourceStepId: string | null  // which journey step provided the answer
}

export interface AggregatedSection {
  sectionId: string
  title: string
  description: string
  steps: AggregatedStep[]
  completionRate: number   // 0-1, how many steps have answers
}

export interface AggregatedReportInput {
  founderName: string
  companyName: string
  industry: string
  sections: AggregatedSection[]
  totalAnswered: number
  totalSteps: number       // always 19
}
```

Use the `sql` tagged template from `lib/db/supabase-sql` (same pattern as founder-reports.ts). Use a service-role client for reads since this runs server-side during report generation.

Handle edge cases:
- User has zero progress rows → return all null answers with completionRate 0
- metadata field is null or missing answer/distilled keys → treat as null answer
- Profile missing name/company → use "Founder" / "Your Venture" defaults

Do NOT filter by stage — pull ALL oases_progress rows and let the step mapping determine which ones are relevant. This ensures steps that span multiple stages are captured.
  </action>
  <verify>
Run: `npx tsc --noEmit` — no TypeScript errors.
Verify the function signature: `aggregateReportData(userId: string): Promise<AggregatedReportInput>`.
  </verify>
  <done>
aggregator.ts exports `aggregateReportData()` that queries oases_progress + profiles and returns structured `AggregatedReportInput` with all 19 steps mapped to 5 sections, null for missing answers.
  </done>
</task>
