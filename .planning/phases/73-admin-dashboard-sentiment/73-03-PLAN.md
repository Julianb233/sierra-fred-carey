  <name>Task 1: Create session detail API and triage API</name>
  <files>app/api/admin/feedback/sessions/[sessionId]/route.ts, app/api/admin/feedback/triage/route.ts, lib/db/feedback-admin.ts</files>
  <action>
Add to `lib/db/feedback-admin.ts`:

1. Export `getSessionDetail(sessionId: string)`:
   - Query `feedback_sessions` for the session row.
   - Query `feedback_signals` where `session_id` matches, ordered by `created_at` asc.
   - Query `fred_episodic_memory` where `session_id` matches, ordered by `created_at` asc (to get the full conversation).
   - Return `{ session: FeedbackSession, signals: FeedbackSignal[], messages: Array<{ role: string, content: string, created_at: string }> }`.
   - Use `createServiceClient()` (admin bypass RLS).

2. Export `updateTriageStatus(id: string, table: 'feedback_signals' | 'feedback_insights', newStatus: string)`:
   - Validate newStatus is in INSIGHT_STATUSES.
   - Update the row's `status` field.
   - If newStatus is 'actioned', set `actioned_at` to now.
   - If newStatus is 'resolved', set `resolved_at` to now.
   - Return the updated row.

3. Export `getSessionsWithFeedback(limit?: number)`:
   - Query `feedback_sessions` joined with count of `feedback_signals` per session.
   - Order by `created_at` desc.
   - Include `flagged` field for visual prominence.
   - Return array of sessions with signal counts.

Create `app/api/admin/feedback/sessions/[sessionId]/route.ts`:

1. GET handler:
   - `requireAdminRequest(request)` -- return 401 if denied.
   - Extract `sessionId` from route params.
   - Call `getSessionDetail(sessionId)`.
   - Return JSON response.

Create `app/api/admin/feedback/triage/route.ts`:

1. PATCH handler:
   - `requireAdminRequest(request)` -- return 401 if denied.
   - Parse body: `{ id: string, table: 'feedback_signals' | 'feedback_insights', status: string }`.
   - Validate status is in INSIGHT_STATUSES.
   - Call `updateTriageStatus(id, table, status)`.
   - Return updated row as JSON.
  </action>
  <verify>
  Run `npx tsc --noEmit`. Verify API routes export correct HTTP methods.
  </verify>
  <done>
  - Session detail API returns conversation messages with feedback annotations
  - Triage API accepts PATCH to update status through the workflow
  - Both APIs are admin-auth protected
  </done>

<task type="auto">
  <name>Task 2: Build session drill-down page with sentiment arc and triage UI</name>
  <files>app/admin/feedback/[sessionId]/page.tsx, app/admin/feedback/page.tsx</files>
  <action>
Create `app/admin/feedback/[sessionId]/page.tsx` as a "use client" page:

1. **Header section**:
   - Back link to /admin/feedback.
   - Session ID, user ID, channel, started_at, status.
   - Flagged badge (red) if session is flagged, with flag_reason tooltip.

2. **Sentiment arc** (visual timeline):
   - A horizontal bar/line showing sentiment scores over time.
   - Each message gets a dot on the timeline.
   - Color: green (positive), gray (neutral), orange (negative), red (frustrated).
   - Use simple div-based visualization (dots positioned proportionally along a horizontal line).
   - If sentiment degraded sharply (flagged), highlight the degradation section in red.
   - Below the arc, show the trend label: "improving", "stable", "degrading", "spike_negative".

3. **Conversation replay** (main content):
   - Render each message in a chat-like layout:
     - User messages on the right (gray background).
     - FRED responses on the left (slightly different background).
   - For each message, if there's a corresponding feedback_signal:
     - Show inline annotation: thumbs icon, category badge, sentiment badge.
     - If category is 'coaching_discomfort', show a blue "Coaching Discomfort" badge with a tooltip: "User reacted to FRED challenging their assumptions -- this is FRED working as designed."
   - Timestamps between messages.

4. **Triage panel** (sidebar or bottom section):
   - Current status displayed as a badge with the current workflow step highlighted.
   - Status workflow buttons: a row of buttons for each valid next status.
     - From "new": can go to "reviewed"
     - From "reviewed": can go to "actioned"
     - From "actioned": can go to "resolved"
     - From "resolved": can go to "communicated"
   - Clicking a status button calls PATCH `/api/admin/feedback/triage` with the insight/session ID and new status.
   - Show confirmation after update.
   - For the triage workflow, use the `feedback_insights` table if an insight exists for this session, otherwise create one on first status change.

5. **Signals summary** (below conversation):
   - Table of all feedback signals for this session.
   - Columns: Time, Type, Rating, Category, Sentiment Score, Confidence, Comment.

- Add a "Sessions" tab or section that lists sessions from `getSessionsWithFeedback()`, showing: session ID, user, channel, message count, sentiment trend, flagged status.
- Flagged sessions should have a red left border or red badge to draw attention.

Style: Match existing admin dark mode patterns. Use `@/components/ui/badge` for status badges if available, otherwise use styled spans. Sentiment colors: positive=#22c55e, neutral=#6b7280, negative=#f97316, frustrated=#ef4444.
