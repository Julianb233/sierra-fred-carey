  <files>lib/oases/stage-topics.ts, lib/oases/intent-classifier.ts</files>
  <action>
    Create `lib/oases/stage-topics.ts`:
    - Export `STAGE_TOPIC_MAP: Record<OasesStage, { keywords: RegExp[]; descriptions: string[] }>` mapping each stage to its allowed conversation topics:
      - **clarity**: profile, who am I, what am I building, idea exploration, problem definition, first chat, basic business fundamentals, self-assessment, reality lens, general startup questions, introductions
      - **validation**: positioning, target market, competitive analysis, customer interviews, value proposition, market research, ICP, demand signals, assumptions testing, product-market exploration
      - **build**: pitch deck, strategy document, business plan, investor readiness assessment, financial model, MVP, product development, go-to-market strategy, unit economics
      - **launch**: fundraising, investor outreach, investor targeting, pitch practice, term sheets, valuation, cap table, due diligence, investor meetings, pitch refinement
      - **grow**: fund matching, Boardy, scaling, hiring at scale, Series B+, team building, international expansion, board management, growth metrics
    - Each keyword entry is a RegExp with word boundaries for accurate matching
    - Export a `getStageForTopic(topic: string): OasesStage | null` helper

    Create `lib/oases/intent-classifier.ts`:
    - Import `STAGE_TOPIC_MAP` and `OasesStage` type
    - Export `classifyIntent(message: string): { detectedStage: OasesStage | null; matchedTopic: string | null; confidence: 'high' | 'medium' | 'low' }`
    - Logic: lowercase the message, test against each stage's keywords in order from highest stage (grow) to lowest (clarity) — this ensures we catch premature topics first
    - If multiple stages match, return the HIGHEST stage (most restrictive) — this is conservative and prevents users from bypassing gates
    - If no keywords match, return `{ detectedStage: null, matchedTopic: null, confidence: 'low' }` — meaning the message is general conversation and should be allowed
    - Confidence levels: 'high' if 2+ keywords from same stage match, 'medium' if 1 keyword matches, 'low' if no match

    IMPORTANT: Use `\b` word boundaries in RegExp to avoid false positives (e.g., "build" in "I'm building my profile" should match clarity, not build stage). Be specific with multi-word phrases like `pitch\s*deck`, `investor\s*readiness`, `fund\s*matching`.

    IMPORTANT: General startup questions like "what should I focus on", "how do I start", greetings, and emotional/wellbeing topics should NOT match any stage (return null) — these are always allowed regardless of stage.
  </action>

    - "I'm feeling overwhelmed" -> detectedStage: null
    - "help me with fundraising" -> detectedStage: 'launch'

<task type="auto">
  <name>Task 2: Create stage validator with redirect messaging</name>
  <files>lib/oases/stage-validator.ts</files>
  <action>
    Create `lib/oases/stage-validator.ts`:
    - Import `classifyIntent` from `./intent-classifier`
    - Import `getStageIndex, getStageConfig, STAGE_ORDER` from `./stage-config`
    - Import `OasesStage` from `@/types/oases`

    Export interface `StageValidationResult`:
    ```typescript
    interface StageValidationResult {
      allowed: boolean
      currentStage: OasesStage
      detectedStage: OasesStage | null
      redirectMessage: string | null  // The gentle redirect message for FRED to use
      currentStageGuidance: string | null  // What FRED should talk about instead
    }
    ```

    Export function `validateStageAccess(message: string, currentStage: OasesStage): StageValidationResult`:
    1. Call `classifyIntent(message)` to get detected stage
    2. If `detectedStage` is null (general conversation) or confidence is 'low', return `{ allowed: true, ... }` — always allow general topics
    3. Compare `getStageIndex(detectedStage)` vs `getStageIndex(currentStage)`
    4. If detectedStage index <= currentStage index, return `{ allowed: true, ... }` — topic is at or before current stage
    5. If detectedStage index > currentStage index, return `{ allowed: false, ... }` with:
       - `redirectMessage`: A warm, mentor-style redirect. Use templates based on the gap:
         - 1 stage ahead: "I can see you're eager to get to {detectedStageName} — that's a great sign. But right now, let's make sure your {currentStageName} foundation is solid. {currentStageGuidance}"
         - 2+ stages ahead: "We'll absolutely get to {detectedStageName} — but we need to walk before we run. You're in {currentStageName} right now, and getting this right is what makes everything downstream work. {currentStageGuidance}"
       - `currentStageGuidance`: Pull from stage config. Format: "In {currentStageName}, focus on: {steps as comma-separated list}. Which of these would you like to tackle?"

    Export function `buildStageGatePromptBlock(currentStage: OasesStage): string`:
    - Returns a prompt block to inject into FRED's system prompt that:
      1. States the user's current Oases stage and what topics are appropriate
      2. Lists the topics that belong to LATER stages (with stage names) so FRED knows what to redirect
      3. Provides redirect language templates FRED should use
      4. Emphasizes: "Do NOT help with later-stage topics even if the founder insists. Redirect warmly."
      5. Includes the current stage's steps as the conversation agenda
    - Format as markdown sections for clean prompt injection

    IMPORTANT: The redirect tone must match Fred Cary's voice — direct but warm, never condescending. "Let's nail X first" not "You can't do that yet." Frame it as strategic sequencing, not restriction.

    IMPORTANT: `buildStageGatePromptBlock` is the key function — it generates the prompt instructions that will be injected into FRED's system prompt in Plan 02. It must be comprehensive enough that the LLM understands the gating rules without any additional code-level enforcement.
  </action>
