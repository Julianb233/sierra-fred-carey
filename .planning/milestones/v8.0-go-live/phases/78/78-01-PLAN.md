<task type="auto">
  <name>Task 1: Wire progress computation to journey_steps table and enhance progress API</name>
  <files>
    lib/oases/progress.ts
    app/api/oases/progress/route.ts
  </files>
  <action>
    The existing `lib/oases/progress.ts` computes progress from the static 14-step STAGE_CONFIG. The `journey_steps` table (migration 20260312000002) has ~120 detailed steps seeded per stage. Update the progress computation to:

    1. In `getUserOasesProgress()`, query the `journey_steps` table to get per-stage step counts:
       ```sql
       SELECT stage, count(*) as total_steps FROM journey_steps WHERE active = true GROUP BY stage
       ```
    2. Query the `oases_progress` table (user-scoped) to count completed steps per stage:
       ```sql
       SELECT stage, count(*) as completed FROM oases_progress WHERE user_id = $1 GROUP BY stage
       ```
    3. Compute per-stage completion: `stepsCompleted / stepsTotal` for each stage.
    4. Compute overall `journeyPercentage` as weighted average across all 5 stages (equal weight per stage = 20% each, so each stage contributes `(stepsCompleted / stepsTotal) * 20` to total).
    5. Determine `currentStage`: the first stage where `stepsCompleted < stepsTotal`. If all complete, currentStage = "grow".
    6. Determine `canAdvance`: true only if current stage's stepsCompleted equals stepsTotal.
    7. Return the enriched OasesProgress object with per-stage completedStepIds from oases_progress rows.

    For the progress API route (`app/api/oases/progress/route.ts`):
    - Ensure it returns `{ success: true, data: OasesProgress }` with the enriched data.
    - Add a `Cache-Control: private, max-age=30` header so progress updates within 30s on the client.

    IMPORTANT: Keep backward compatibility with the existing StageConfig steps array in stage-config.ts. The 14 high-level steps remain as the UI-facing checklist in StageDetailModal. The journey_steps table provides the granular tracking denominator for progress percentage.

    Approach: The journey percentage should blend both systems — the 14 STAGE_CONFIG steps drive the checklist UI, while the journey_steps table drives the percentage bar. Add a new function `getDetailedProgress()` that queries journey_steps for the percentage, and keep existing `getUserOasesProgress()` for the checklist UI. Merge both in the API response.
  </action>

    - The OasesProgress type still has all required fields (currentStage, stageIndex, stages[], journeyPercentage, canAdvance)
  </verify>
