     - Validates each step completes (checks for expected DOM elements or API responses)

     - Cleans up test account after run

  2. Create `tests/e2e/onboarding-journey-smoke.spec.ts` -- Playwright E2E that:

     - Validates Oases journey visualizer renders with correct initial stage
     - Checks that stage gating prevents access to locked features
     - Uses the existing Playwright config (testDir: ./tests/e2e)
     - Skip tests gracefully if Supabase env vars are missing (like existing e2e tests do)

  3. Create `tests/e2e/mobile-voice-continuity.spec.ts` -- Playwright mobile viewport E2E that:

     - Tests voice coaching page renders controls correctly on mobile
     - Validates chat-to-voice UI transition elements are tappable (>= 44px touch targets)
     - Checks no horizontal scroll on chat and voice pages

<task type="auto">
  <name>Task 2: Event feedback widget + iteration SLA dashboard (TEST-03, TEST-04)</name>
  <files>
    components/feedback/event-feedback-widget.tsx

    lib/feedback/iteration-tracker.ts

  <action>
  1. Create `components/feedback/event-feedback-widget.tsx` -- a "use client" component that:
     - Shows a floating button (bottom-right) labeled "Share Feedback" for users with `test_group` containing "event"
     - On click, opens a modal/dialog with:
       a. 5-star rating for overall experience
       b. 5-star rating for FRED specifically
       c. "Would you recommend Sahara?" (Yes/Maybe/No radio buttons)
       d. Free-text "What could be better?" textarea
       e. Free-text "What did you love?" textarea
     - Submits to POST `/api/feedback/event`
     - Shows success toast (sonner) on submit, hides widget for the session
     - Uses shadcn Dialog + Button + Textarea components
     - Accepts `eventName` and `userId` props

  2. Create `app/api/feedback/event/route.ts` -- POST endpoint that:
     - Accepts: { userId, eventName, rating, fredRating, recommend, improvementText, loveText }
     - Validates required fields (userId, rating)
     - Inserts into `event_feedback` table (same schema the existing admin/event-feedback page reads from)
     - Also creates a `feedback_signals` row with channel="event_survey" so the existing feedback pipeline picks it up
     - Returns 201 on success, 400 on validation error
     - Uses createServiceClient() for server-side Supabase

  3. Create `lib/feedback/iteration-tracker.ts` that:
     - Exports `getIterationMetrics(sinceDays: number)` which queries:
       a. feedback_signals created in the window grouped by status (new, triaged, resolved)
       b. prompt_patches created in response to feedback, with avg time from signal to patch
       c. Percentage of signals resolved within 48 hours (SLA compliance)
     - Exports `getIterationTimeline(sinceDays: number)` which returns daily counts of signals received vs resolved
     - Uses createServiceClient()

  4. Create `app/admin/testing/iteration-dashboard.tsx` -- "use client" component that:
     - Fetches iteration metrics from a new GET handler (add to existing test-accounts route or create separate)
     - Shows 4 stat cards: Total Signals, Triaged %, Resolved in 48h %, Avg Resolution Time
     - Shows a simple table of recent signals with status and time-to-resolution
     - Color-codes SLA compliance: green >= 80%, yellow 50-79%, red < 50%

  5. Update `app/admin/testing/page.tsx` to add a third tab "Iteration" that renders `<IterationDashboard />`.
     Import the new component and add the tab button + conditional render.
  </action>
