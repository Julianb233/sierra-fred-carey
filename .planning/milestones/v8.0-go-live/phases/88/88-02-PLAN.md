<objective>
Build an event conversion analytics dashboard so the team can track the funnel from QR scan to first FRED interaction for the Palo Alto event.

Purpose: The success criteria require tracking "QR scans, signups, onboarding completions, first FRED interactions." PostHog tracks individual events but there's no admin-facing page that shows the funnel. This plan creates a lightweight admin dashboard that queries the profiles table (which has `event_source`, `onboarding_completed`, `journey_welcomed`, `reality_lens_complete` columns) to compute funnel metrics.

Output: Admin page at /admin/event-analytics showing conversion funnel per event slug.
</objective>

  1. Add additional analytics event constants to `lib/event/analytics.ts`:
  ```
  ONBOARDING_COMPLETE: "event_onboarding_completed",
  FIRST_FRED_INTERACTION: "event_first_fred_interaction",
  ```
  These are for tracking (the actual tracking calls will be wired in the existing onboarding and chat flows by checking for `event_source` on the profile).

  2. Create `app/api/admin/event-analytics/route.ts` with a GET handler that:
     - Uses `createServiceClient()` (admin access) to query the `profiles` table
     - Accepts optional `?slug=palo-alto-2026` query param to filter by event
     - Returns funnel metrics by counting profiles with `event_source IS NOT NULL`:
       a. **Total signups**: COUNT of profiles where `event_source = slug`
       b. **Onboarding complete**: COUNT where `event_source = slug AND journey_welcomed = true`
       c. **Reality Lens complete**: COUNT where `event_source = slug AND reality_lens_complete = true`
       d. **First chat**: COUNT where `event_source = slug` AND has at least one row in `chat_messages` table (use a subquery or join — check if `chat_messages` table exists, if not use `fred_memories` or similar)

     - Also return a list of recent signups (last 20) with name, email, created_at, and which funnel stages they've completed
     - Protect with admin auth check (follow pattern from existing admin routes like `app/api/admin/event-feedback/route.ts`)

  For QR scan count: this is tracked via PostHog `event_landing_view` events. Since we can't query PostHog directly from the API without complexity, use the total signups as a proxy, or add a note that QR scans are available in the PostHog dashboard. The admin page can display "Landing Page Views: See PostHog" with a link.

  Keep the query simple — aggregate counts with `.select('id', { count: 'exact', head: true })` pattern for each funnel stage.
  </action>
