<objective>
Wire the event signup flow end-to-end so that scanning the QR code leads to account creation with auto-login and redirect into the guided onboarding journey (welcome -> intake -> Reality Lens).

Purpose: The existing event landing page, signup form, and registration API are built but the flow has two critical gaps: (1) event registration redirects to /dashboard instead of /welcome, bypassing the guided journey; (2) the admin.createUser call doesn't create a client session, so the user would hit the auth gate on /welcome and bounce.

Output: Working end-to-end flow: QR scan -> /event/palo-alto-2026 -> signup -> auto-login -> /welcome -> intake -> Reality Lens
</objective>

<task type="auto">
  <name>Task 1: Fix event registration redirect and auto-login</name>
  <files>lib/event/config.ts, app/api/event/register/route.ts</files>
  <action>
  1. In `lib/event/config.ts`, change `redirectAfterSignup` for the `palo-alto-2026` event from `"/dashboard"` to `"/welcome"`. This ensures event attendees go through the guided onboarding journey (welcome screen -> intake form -> Reality Lens) instead of landing on the raw dashboard.

  2. In `app/api/event/register/route.ts`, after successfully creating the user with `supabase.auth.admin.createUser()` and updating the profile, add a step that signs the user in to create a client-side session. The current flow uses the admin API which doesn't create a session cookie — so when the client redirects to /welcome, the auth gate bounces them back to /get-started.

  Approach: After the admin createUser succeeds and the profile upsert completes, use a **non-admin** Supabase client (from `createClient`) to call `signInWithPassword({ email, password })`. Return the session data (access_token, refresh_token) in the response alongside the existing `redirectTo` field.

  3. In `components/event/event-signup-form.tsx`, after receiving the successful response from `/api/event/register`, set the Supabase session using the returned tokens before redirecting. Use `supabase.auth.setSession({ access_token, refresh_token })` from the browser client. This ensures the redirect to /welcome finds an authenticated user.

  Also update the form's `onSuccess` callback to handle the new response shape (which now includes session tokens).

  Important: Do NOT change the admin createUser flow itself — that's needed for email auto-confirmation. The signInWithPassword is an additional step after user creation.

  4. In `lib/event/config.ts`, also set `journey_welcomed: false` explicitly in the profile upsert (already present in the register route, verify it's there) so the /welcome page shows the welcome screen instead of redirecting to dashboard.

  Note: The register route already sets `onboarding_completed: true` — this is correct because event signup IS the "onboarding" (stage/challenge selection happens on /welcome via intake form instead).
  </action>
