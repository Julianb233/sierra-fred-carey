    **VOICE-01: Chat Freezing/Response Chopping**

    The chat freezing issue manifests when SSE streaming gets stuck or the response is chopped mid-stream. Investigate and fix:

    1. In `lib/hooks/use-fred-chat.ts`:
       - The `sendingRef.current` guard (line ~269) prevents duplicate sends but could cause "freezing" if it's never reset after an error. Verify the `finally` block at line ~648 always resets `sendingRef.current = false`.
       - Check the abort controller cleanup: if `abortControllerRef.current` is not properly nulled after completion, subsequent sends might be aborted immediately.
       - The auto-retry logic (line ~599) removes partial streaming messages and retries once. If both attempts fail, verify the error message is shown and `sendingRef.current` is reset.
       - Add a safety timeout: if a stream hasn't received any data within 30 seconds of connecting, abort and show a timeout error. This prevents indefinite "connecting..." states that users perceive as freezing.

    2. In `lib/hooks/use-fred-chat.ts`, the streaming message update pattern:
       - Each token event creates a new message object via `setMessages(prev => prev.map(...))`. On long responses this is O(n) per token where n is total messages.
       - This is unlikely to cause freezing but could cause jank on mobile. Consider accumulating tokens in a ref and flushing to state on a requestAnimationFrame cadence (every ~16ms) instead of on every token event. This reduces React re-renders from hundreds to ~60/sec.

    3. In `app/api/fred/chat/route.ts`:
       - Read the streaming response logic
       - Verify the SSE stream properly sends `event: done` at the end of every response
       - Verify error events are sent if the LLM stream fails mid-response
       - If the route uses `ReadableStream` directly, verify it handles backpressure (the client might not be reading fast enough on mobile)
       - Check if there's a max response time or timeout that could cause mid-response chopping

    The key fix for VOICE-01 is the streaming timeout + token batching. The existing retry logic handles transient failures, but indefinite hangs need a timeout.
